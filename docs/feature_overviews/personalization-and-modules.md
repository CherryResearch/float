personalizing float is about shaping how it behaves without forcing every change through the same mechanism. Some customization belongs in the system prompt, some belongs in workflows, and some belongs in modules or add-ons.

The simplest layer is the base system prompt, which defines float's default voice and operating expectations. Workflows sit one level above that: they are lightweight profiles that tell Float how to approach a run, such as favoring more planning or a shorter execution loop. Modules and add-ons extend the actual capability surface. In float's terms, an add-on is a package of skills, tools, and assets that can be loaded together. Themes are the visual customization layer: Float lets a user create a theme from Settings, choose values for the eight available color slots, save it, rename it, edit it later, or delete it, with user-created themes stored under `data/themes/`.

For the user, the intended model is: keep the built-in tools stable, adjust the prompt when you want to change overall behavior, choose workflows when you want a different task posture, use modules and add-ons when you want to introduce a new package of capabilities, and use themes when you want to change Float's visual identity without touching tracked source files. For adjacent runtime and tool detail in the public docs, see `docs/feature_overviews/models-and-runtime-modes.md` and `docs/feature_overviews/tools-and-actions.md`.

float ships with a small set of built-in themes and paired dark/light presentation. If a user wants a personal variation rather than editing the built-in files, the intended flow is to create a custom theme in Settings, choose the palette slots there, and keep that saved theme in `data/themes/` instead of in the tracked frontend source tree.
